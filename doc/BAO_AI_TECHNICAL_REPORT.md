## 1. 概要

この文書は、Bao la Kiswahili のブラウザ向けコンピュータ対戦AIを強化した過程と、そこで得られた技術的知見を第三者が参照できる形でまとめたレポートである。

対象は、`public/`で動作する軽量なWeb実装である。ニューラルネットワークや大規模な外部計算資源は使わず、次の構成で強化した。

- 既存ルールエンジンによる合法手生成と局面遷移
- Bao固有特徴量を加えた評価関数
- Iterative Deepening、Alpha-Beta、Transposition Table、PVS、Quiescence Search
- Web Workerによる探索処理の分離
- seed付き自己対戦、paired opening、戦術回帰テスト
- 重みチューニングとMCTS比較研究

結論として、現時点で最も有効だったのは「Bao固有の局面評価を慎重に追加したAlpha-Beta系探索」である。MCTS、局面カテゴリ別評価`bao-v2`、適応的探索予算、追加の自己対戦チューニングv2はいずれも実験基盤としては有用だったが、既定AIへ採用するだけの統計的優位は示さなかった。

このレポートでは、成功した変更だけでなく、不採用にした試行、測定上の注意、今後対人ログから改善するための方針も記録する。

### 1.1 この文書の読み方

短時間で全体を把握したい場合は、まずこの章と「13. 主要な知見」を読む。実装方式を追う場合は「5. 評価関数」「6. 探索」「7. Web Worker化」を、実験設計を再利用したい場合は「8. 測定方法」「10. 自己対戦チューニング」を読むとよい。

この文書の数値は、特に断りがない限り、このリポジトリの実装、当時のNode.js実行環境、記載したseed、探索条件で得た結果である。別実装や別端末で同じ強さを保証するものではない。

### 1.3 採用判定の基準

採用判断では、短い自己対戦の勝率だけを使わなかった。原則として、候補は次の条件を満たす必要がある。

- 500局以上の検証でWilson 95%下限が50%を超える
- Phase 6以降の戦術回帰を悪化させない
- 時間制限付き試験で平均・最大思考時間やtimeoutが実用上悪化しない
- 変更後もWorker経由のUI応答性と合法手検証を維持する

この基準により、勝率だけを見ると有望に見える候補でも、統計的優位や戦術品質を示せない場合は既定AIへ反映しなかった。

## 2. 開発上の制約

Bao AIはWebブラウザ上で遊べるゲームの一部として実装されている。そのため、強さだけでなく、応答性、再現性、保守性を同時に満たす必要があった。

主な制約:

| 制約 | 内容 |
| --- | --- |
| 実行環境 | 静的サイトとしてCloudflare Pagesへ公開する |
| UI応答性 | 探索中も画面操作や描画を止めない |
| 再現性 | 変更前後の強さをseed付き自己対戦で比較できる |
| 説明可能性 | 評価値や採用判断を後から説明できる |
| 保守性 | ルール判定は`engine.js`へ集約し、AIが盤面を直接改変しない |
| 公開品質 | 採用見送りや失敗した試行も記録し、後続開発の判断材料にする |

## 3. Bao AI化で難しかった点

Bao la Kiswahiliでは、単純な石数差や直後の捕獲数だけでは局面の良し悪しを評価しにくい。特に次の要素がAI化を難しくした。

| 要素 | AI実装上の難しさ |
| --- | --- |
| namua / mtaji | reserveを持つ段階と、盤上の石だけで戦う段階で評価基準が変わる |
| relay sowing | 1手の内部で捕獲や播種が連鎖し、着手前の局所情報だけでは結果を見誤る |
| nyumba | 維持すべき局面と、崩して使うべき局面がある |
| 前列 | 捕獲力、防御、敗北耐性に直結する |
| 反撃 | 大量捕獲の直後に前列を崩される局面がある |
| 終盤 | 早い勝利だけでなく、避けられない敗北を遅らせる評価が必要になる |

そのため、本AIでは「探索を深くする」前に、葉局面をBaoらしく評価できることを重視した。

### 3.1 用語メモ

| 用語 | この文書での扱い |
| --- | --- |
| namua | reserveから石を投入できる段階 |
| mtaji | reserveがなくなり、盤上の石だけで着手する段階 |
| nyumba | Bao固有の家。維持価値と使用価値が局面により変わる |
| relay sowing | 1手の内部で播種と捕獲が連鎖する動き |
| 前列 | 相手陣に接する列。捕獲、守備、敗北耐性に強く関わる |

## 5. 評価関数

現在の既定評価`bao`は、旧評価を完全に置き換えるのではなく、旧評価にBao固有特徴量の差分を加える方式である。

```text
Bao評価 = legacy評価 + Σ((自分の特徴量 - 相手の特徴量) × フェーズ別重み)
```

全面置換ではなく補正方式にした理由は、Bao固有特徴量だけで旧評価を置き換えた初期試行が大きく弱化したためである。標準初期局面、深度3、20局の試験では、Bao特徴量による全面置換は旧評価に0勝20敗だった。

主な特徴量:

| 特徴量 | 意図 |
| --- | --- |
| `boardSeeds` | 自陣の総石数 |
| `frontSeeds` | 前列の石数 |
| `frontOccupied` | 前列に石がある穴数 |
| `frontConnections` | 前列の連結性 |
| `reusablePits` | 再び動かせる穴 |
| `mobility` | pass以外の合法手数 |
| `captureMoves` | 捕獲合法手数 |
| `maxCapture` | 合法手を実際に適用したときの最大捕獲量 |
| `relayShape` | relay sowingを含む捕獲形の近似 |
| `frontSafety` | 前列崩壊リスクの抑制 |
| `houseValue` | nyumbaの利用・維持価値 |
| `reserveEfficiency` | namuaでのreserve投入効率 |
| `transitionShape` | mtaji移行時の形 |
| `tempo` | 手番差 |

重要だったのは、`maxCapture`を対面穴の石数で近似せず、`engine.js`で合法手を実際に適用し、relay sowing中の全`capture`イベントを合計したことである。

観測結果:

| 試行 | 条件 | 結果 |
| --- | --- | ---: |
| 近似捕獲量を使った補正評価 | paired opening、200局 | 123勝77敗、61.5% |
| 実遷移捕獲量、重み8 | paired opening、200局 | 135勝65敗、67.5% |

一方で、評価コストは上がった。旧評価の平均4.11msに対して、Bao評価は平均20.72msになった。評価精度と探索速度は交換条件にあり、後続の探索改善とセットで扱う必要があった。

現在の既定重み:

| 特徴量 | namua | mtaji |
| --- | ---: | ---: |
| boardSeeds | 1 | 2 |
| frontSeeds | 1 | 2 |
| frontOccupied | 5 | 7 |
| frontConnections | 3 | 4 |
| reusablePits | 3 | 5 |
| mobility | 2 | 3 |
| captureMoves | 3 | 5 |
| maxCapture | 8 | 8 |
| relayShape | 1 | 1 |
| frontSafety | 8 | 12 |
| houseValue | -7 | 0 |
| reserveEfficiency | 1 | 0 |
| transitionShape | 2 | 0 |
| tempo | 2 | 3 |

この表は説明用の写しであり、正本は実装内の`EVALUATION_WEIGHTS`である。

## 6. 探索

既定探索`phase2`では、Alpha-Beta系探索に次の改善を加えた。

- Iterative Deepening
- Transposition Table
- Principal Variation Search
- 前回反復深化の最善手優先
- killer move
- 即時勝利、捕獲量、静的評価による着手順序
- 捕獲局面向けのQuiescence Search
- 終局距離評価

Phase 2の検証結果:

| 指標 | 結果 |
| --- | ---: |
| 450ms比較の平均探索ノード数 | 353から717へ、2.03倍 |
| 代表中盤局面 | 深度6および深度7へ到達 |
| Phase 1探索との200局比較 | 126勝74敗、63.0% |

注意点として、2.03倍にはQuiescence Searchのqnodeを含む。また、捕獲延長が入るため、単純に同一探索木で比較しているわけではない。

Quiescence Searchは有効だったが、深くしすぎると悪化した。初期の深度4設定では代表局面の深度1だけで約500ノードを消費し、戦術選択も悪化した。そのため既定深度は1にした。

終局値には探索plyを反映し、より早い勝利と、避けられない敗北を遅らせる選択を可能にした。この設計では同じ盤面でもrootからの距離が違うと値が変わるため、Transposition Tableの内部キーには局面キーだけでなく探索plyも含めた。

## 7. Web Worker化

探索時間を伸ばすとメインスレッドが止まるため、Phase 3で探索をWeb Workerへ分離した。

実装した安全策:

- 探索ごとにWorkerを生成し、完了または中止時に`terminate()`する
- 新規対局時は古い探索を即時中止する
- 探索世代IDを使い、古い結果を破棄する
- 探索開始局面キーと現在局面キーを照合する
- 着手適用直前に合法手であることを再検証する
- Worker非対応時は同期探索へフォールバックする

Workerへcancelメッセージを送る方式は採用しなかった。同期JavaScript探索中のWorkerはイベントループへ戻るまでcancelメッセージを処理できないため、即時中止には`terminate()`が確実だった。

Node.jsの実Worker統合テストでは、探索中もメイン側5msタイマーが継続すること、1秒探索を20ms後にterminateした場合に結果が配送されないことを確認した。

## 8. 測定方法

この開発では、自己対戦勝率だけでなく、再現性と戦術回帰を重視した。

基本方針:

- 乱数生成器に明示的なseedを渡す
- 固定深度試験では`--time-limit 0`で時間切れを無効化する
- 時間制限試験と固定深度試験を区別する
- 偶数局と次の奇数局で同じランダム序盤局面を共有する
- 2局目ではSouth/Northを交換する
- 候補比較では同一seed、同一opening条件を使う

この2局1組の方法をpaired openingと呼ぶ。決定論的AI同士では、初期局面だけを繰り返すと実質的に同じ対局になりやすいため、ランダム序盤局面を共有したうえで先後を入れ替えた。

記録する主な項目:

| 項目 | 理由 |
| --- | --- |
| 勝敗、先後別勝率 | 先手・後手の偏りを見る |
| 平均手数 | 極端な短期決着や引き延ばしを確認する |
| 平均・最大思考時間 | UI応答性への影響を見る |
| 探索ノード数、到達深度 | 探索効率を見る |
| timeout数 | 時間制限下の安定性を見る |
| 戦術回帰結果 | 勝率では見えない局面品質を見る |
| Wilson 95%下限 | 小標本の偶然を採用しない |

### 8.1 Phase 0基準成績

| 条件 | 結果 |
| --- | ---: |
| 100局、seed 20260706、hard対normal、固定深度2 | hard 98勝2敗、勝率98.0% |
| hard平均 / 最大思考時間 | 4.29ms / 44.55ms |
| hard平均探索ノード数 | 22 |

### 8.2 測定値の扱い

- 固定深度試験は、評価関数や探索の差を再現しやすい
- 時間制限試験は、実ブラウザに近いが端末負荷やJITの影響を受ける
- 小標本の勝率は候補選別には使えるが、採用根拠にはしない
- 途中成績は探索条件が違う場合があるため、異なるPhase間で直接比較しない
- 正式な数値の正本は`AI_BENCHMARK.md`と`artifacts/`のJSON出力とする

## 9. 戦術回帰

自己対戦勝率だけでは、Baoらしい局面判断の悪化を見落とす可能性がある。そのため、固定局面の戦術回帰を整備した。

現在のカテゴリ:

| カテゴリ | 意図 |
| --- | --- |
| `forced-win` | 即時勝利を逃さない |
| `capture-relay` | relay sowingを含む複数捕獲を読む |
| `nyumba-destruction` | nyumbaを崩した方が得な局面で使用する |
| `nyumba-preservation` | 短期捕獲よりnyumba温存が得な局面で維持する |
| `namua-endgame` | 最後のreserve投入後、mtajiで戦える前列形を残す |
| `two-ply-counter-avoidance` | 大きな即時反撃を許す手を避ける |
| `mtaji-endurance` | mtajiで捕獲より可動性と前列維持を優先する |

診断出力:

```sh
BAO_TACTICAL_DIAG=1 node test/tactical.test.js
```

候補重みを検証する場合:

```sh
BAO_AI_WEIGHTS=artifacts/candidate.json \
  BAO_TACTICAL_DIAG=1 node test/tactical.test.js
```

診断出力はJSON Linesで、カテゴリ、選択手、root評価、着手後評価、探索統計を含む。

## 10. 自己対戦チューニング

Phase 4では、評価関数の重みを分離し、自己対戦によるチューニングを行った。

実装した内容:

- 候補重みのseed付き突然変異
- hill climbing
- 現行AIと直近世代を相手にするアーカイブ対戦
- opening plies 4/8/12による途中局面開始
- 学習seedと検証seedの分離
- Wilson 95%下限による判定
- successive halving
- JSON候補保存と再読込

初期実装では、候補ごとに異なるseedとopeningを割り当てていた。これは候補の強さと開始局面の差を混同するため、公平な比較ではなかった。修正後は、同一世代の候補を同じseed、同じpaired openingで比較するようにした。

主な失敗例:

| 試行 | 学習 | holdout | 判断 |
| --- | --- | --- | --- |
| 3世代×8候補、変異幅4 | 世代2で75%、世代3で62.5% | 40局20勝20敗 | 比較条件の欠陥を発見。不採用 |
| 公平化、深度2、2世代×6候補 | 世代2で75% | 40局21勝19敗 | Wilson下限37.5%で不採用 |
| `houseValue: 5`周辺、深度2 | 62.5% | 60局31勝29敗 | Wilson下限39.3%で不採用 |
| mtaji限定変異、深度2 | 75% | 60局29勝31敗 | holdoutで悪化。不採用 |

最終的に、複数seed版successive halvingで`namua.houseValue: -7`候補が残った。未使用seed、opening plies 4/8/12を含む500局で279勝221敗、勝率55.8%、Wilson 95%下限51.4%となり、Phase 4の採用条件を満たした。戦術回帰も通過したため、この重みを既定値へ反映した。

この結果から、小規模自己対戦の高勝率は採用根拠にせず、holdout、複数seed、Wilson下限、戦術回帰を併用する必要があると分かった。

## 11. MCTS比較研究

Phase 5では、Alpha-Beta系探索に対してMCTSを比較した。

試した内容:

- UCT
- random / capture / balanced / evaluation プレイアウト
- root選択のvisits / value切替
- evaluation / terminal / fast-terminal報酬
- root子ノードへのstatic prior
- root候補制限
- 浅いPhase 2探索を使ったhybrid候補制限

結果として、MCTS単体は既定のPhase 2探索を上回らなかった。Phase 2探索をroot候補制限に使うhybrid検証でも改善は確認できなかったため、Phase 5ではMCTSを不採用とした。

この結果は「BaoにMCTSが向かない」という一般結論ではない。少なくともこの実装、時間制限、プレイアウト設計、評価関数では、軽量ブラウザAIとしてAlpha-Beta系探索の方が有利だった、という意味である。

## 12. Phase 7以降の追加実験

Phase 6以降では、既定AIをさらに改善するために、戦術局面の拡充、評価関数v2、適応的探索予算、自己対戦チューニングv2を実施した。

### 12.1 戦術局面セットの拡充

Phase 6では、戦術回帰をカテゴリ駆動に整理し、`forced-win`、`capture-relay`、`nyumba-destruction`、`nyumba-preservation`、`namua-endgame`、`two-ply-counter-avoidance`、`mtaji-endurance`を追加・整理した。

この段階で、以後の評価候補は自己対戦だけでなく`tactical=7/7`を満たすことを採用条件にした。

### 12.2 評価関数v2

Phase 7では、既存特徴量を局面カテゴリ別に重み補正する`bao-v2`を実装した。

導入したカテゴリ:

- `namua-opening`
- `namua-midgame`
- `namua-endgame`
- `mtaji-attack`
- `mtaji-endurance`
- `mtaji-closing`
- `mtaji-balanced`

`bao-v2`は戦術回帰を通過し、診断APIや候補比較ツールとしては有用だった。しかし、500局longrunでは259勝241敗、score 51.8%、Wilson 95%下限47.4%、`tactical=7/7`で、統計的優位には届かなかった。そのため既定評価には採用しなかった。

この結果から、既存特徴量をカテゴリ別に重み補正するだけでは、既定`bao`を明確に上回る改善にはならない可能性が高いと判断した。

### 12.3 適応的探索予算

Phase 8では、局面の複雑度に応じて探索時間を調整する適応的探索予算を試した。

使った指標:

- 合法手数
- 捕獲手数
- 最大捕獲量
- 前列占有
- phaseとreserve残量

hardでは、固定予算内に収めると時間改善が小さく、勝敗悪化の疑いを消せなかった。expertでは、早期終了ありでは時間短縮したが勝敗悪化の反例があり、早期終了なしでは平均・最大思考時間が悪化した。

最終確認:

| 条件 | 適応予算 | 固定予算 | 判定 |
| --- | --- | --- | --- |
| expert 1500ms | 4勝4敗8分、平均1536.2ms、最大2038.8ms、timeout 127 | 4勝4敗8分、平均1442.7ms、最大1531.7ms、timeout 127 | 時間悪化。不採用 |

既定UIはhard/expertとも固定予算を維持した。ただし、`adaptiveComplexity`や比較ツールは診断用に残した。

### 12.4 自己対戦チューニングv2

Phase 9では、Phase 4のsuccessive halvingを発展させ、戦術回帰ペナルティ、複数基準AI、局面カテゴリ別成績を評価に組み込んだ。

結果:

| 候補 | 500局longrun | Wilson 95%下限 | 判定 |
| --- | ---: | ---: | --- |
| `candidate-01` | 264勝236敗、52.8% | 48.4% | 不採用 |
| `candidate-02` | 264勝236敗、52.8% | 48.4% | 不採用 |
| `candidate-11` | 269勝231敗、53.8% | 49.4% | 不採用 |

`candidate-11`は採用目安の272勝に3勝足りなかった。さらに本格探索では候補24、`round-games=2,6,20`、`round-repeats=3,2,1`、validation 48局で候補を絞ったが、finalists 3件はいずれもvalidationで50%を下回り、promote候補は出なかった。

Phase 9では、既定`bao`を統計的に上回る重み候補は得られなかった。

## 13. 主要な知見

この章では、実装固有の観測と、他のゲームAI開発にも転用しやすい教訓を分けて読む必要がある。特に勝率やWilson下限の値はこのリポジトリの条件に依存する。一方で、候補比較の公平性、holdoutの必要性、戦術回帰の併用は、他の決定論的ゲームAIにも適用しやすい。

### 13.1 大量捕獲は単独では十分な評価指標にならない

Baoでは、大きく捕獲できる手が常に良いとは限らない。捕獲後の前列形、relay sowing、相手の反撃、nyumbaの扱いが次の局面に影響する。捕獲量やrelay、nyumbaを単純に大きく評価した初期候補は旧評価に大敗した。

ただし、「大量捕獲は悪い」と一般化すべきではない。現在の特徴量定義と探索深度では、捕獲量単独では不十分だった、という結果である。

### 13.2 1手の内部遷移をルールエンジンで実行して測る価値があった

`maxCapture`を対面穴の石数で近似するより、合法手を実際に適用してrelay終了までの捕獲を合計した方が有効だった。1手の内部遷移が長いゲームでは、着手前の局所情報だけでなく、手の完了結果を特徴量に使う価値がある。

### 13.3 小標本の自己対戦勝率は危険である

学習75%の候補がholdoutで20勝20敗になった例や、mtaji限定で学習75%だった候補がholdoutで29勝31敗へ悪化した例があった。自己対戦チューニングでは、seed、opening、先後、holdout、Wilson下限を設計しないと、偶然や局面差を強さと誤認しやすい。

### 13.4 候補ごとに異なる開始局面を使うと比較にならない

初期チューナーでは、候補ごとに異なるseedとopeningを使っていた。この条件では、重み候補の差と開始局面の差を分離できない。以後は同一seed、同一paired openingで候補を比較するようにした。

### 13.5 重み空間には平坦な領域が広い

単一重みの小変更では着手系列が変わらない候補が多かった。legacy評価を基礎点としているため、補正値が葉評価を変えてもrootの最善手が変わらない場合がある。今後は重み値だけでなく、候補による着手変更率も測る価値がある。

### 13.6 採用見送りになった実験にも価値がある

`bao-v2`、適応的探索予算、Phase 9チューニングv2はいずれも既定採用には至らなかった。しかし、診断API、候補比較ツール、longrun分割実行、decisionツールは、今後の対人ログ改善や局面カテゴリ分析に再利用できる。

## 14. 現在の限界

現時点のAIには次の限界がある。

| 限界 | 内容 |
| --- | --- |
| 対人ログ不足 | 人間が実際に違和感を持つ局面が十分に集まっていない |
| 戦術局面の偏り | 手作業で作った局面に依存している |
| 評価関数の表現力 | 既存特徴量と重み補正だけでは改善が頭打ちになっている可能性がある |
| 自己対戦依存 | 同じAI系統同士の対戦であり、人間らしい弱点を十分に反映しない |
| 時間制限試験の非決定性 | CPU負荷やJITにより到達深度と勝敗が変化し得る |
| ブラウザ実測 | 開発環境では実ブラウザfpsを自動測定していない |

特に、Phase 7からPhase 9では、既定`bao`を明確に上回る候補が得られなかった。今後は単に重み探索を続けるより、対人ログや失敗局面から新しい戦術局面を作り、評価関数または探索順序の不足を特定する方が有効と考えられる。

## 15. 今後の改善方針

次の段階では、対人ログによる改善ループを中心にする。

推奨する流れ:

1. 対局中にAIの違和感がある手を記録する
2. その局面を匿名化されたJSONとして保存する
3. 人間視点の期待手、避けるべき手、理由を付ける
4. 戦術回帰またはベンチマーク開始局面へ変換する
5. 現行AIが失敗するか確認する
6. 評価関数、探索順序、戦術カテゴリのどれで改善すべきか分類する
7. 自己対戦、戦術回帰、時間制限試験で採用判断する

ログ収集時の注意:

- 明示的な同意なしに個人情報を保存しない
- 操作ログを必要以上に保存しない
- 公開する局面データから個人を特定できる情報を除く
- 勝敗だけでなく、人間がなぜ違和感を持ったかを残す
- 対人由来の改善でも、自己対戦と戦術回帰で副作用を確認する
