# Bao AI 局面監査・対人レビュー手順

Version: 0.3.0  
更新日: 2026-07-13

## 1. 目的

実際の対局で得たAI着手を再現可能な診断局面として保存し、探索改善、評価改善、戦術回帰の候補へ変換する。

この手順は、レビュー担当者がBaoの熟練者であることを前提としない。人間が「この手は弱い」「期待手はこれだ」と判断できない場合でも、観測事実、探索深度差、評価値、自己対局結果を記録することで進められる。

Phase 10は次の2段階に分ける。

| 段階 | 目的 | Baoの高い棋力 |
| --- | --- | --- |
| Phase 10A: 機械的局面監査 | 局面保存、再現、深度比較、客観的異常の抽出 | 不要 |
| Phase 10B: 棋力レビュー | 期待手、避ける手、Bao上の理由を確認し、戦術回帰へ採用する | 必要または外部確認が必要 |

Phase 10Aで未判定の局面が残っても問題はない。根拠が不十分な局面は正式な戦術回帰や評価関数変更へ使わず、将来のレビュー候補として保持する。

## 2. 基本原則

- 初心者は最善手を断定しない。
- 「弱そう」という印象より、観測できた事象を記録する。
- 未判定を失敗扱いしない。
- 期待手とBao上の理由が未確認のfixtureは正式な回帰へ追加しない。
- 1局だけを根拠に評価関数を変更しない。
- 自己対局は絶対的な正解判定ではなく、退行検出と候補比較に使う。
- 熟練者レビューは、自動診断で絞り込んだ少数局面へ集中させる。

## 3. 保存とプライバシー

AI診断記録は利用者がボタンを押した場合だけ、ブラウザのlocalStorageへ保存される。

保存する情報:

- AIが考える前の盤面
- AIが選択した着手
- AI難易度と評価profile
- 探索深度、node数、timeoutなどの許可された探索統計

保存しない情報:

- 自由記述
- 氏名、対局者名
- 端末やブラウザの識別子
- 保存時刻
- AI着手以外の連続操作ログ

記録は最大50件で、外部へ自動送信されない。「記録を削除」またはブラウザのサイトデータ削除機能から消去できる。

診断JSONをファイルへ保存する際は、利用者の端末のローカル日時をファイル名にだけ使用する。JSON本文へ保存時刻は追加しない。

## 4. 診断ボタン

「AI改善用診断」には次の4つのボタンがある。

| ボタン | 対象 | 動作 | 主な用途 |
| --- | --- | --- | --- |
| 現在局面を保存 | ボタンを押した時点の盤面 | JSONファイルを端末へ保存 | 盤面だけを単発で記録する |
| AIの手を記録 | 直前のAI着手前の盤面、選択手、探索統計 | localStorageへ追加 | 後で調べたいAI判断を蓄積する |
| 記録を保存 | localStorageに蓄積した全記録 | JSONファイルを端末へ保存 | 複数局面をまとめて取り出す |
| 記録を削除 | localStorageに蓄積した全記録 | 確認後に削除 | 端末内の記録箱を空にする |

### 4.1 現在局面を保存

ボタンを押した時点の盤面を、単一の診断snapshotとして保存する。

ファイル名:

```text
bao-position-YYYYMMDD-HHMMSS.json
```

- AI着手後に押した場合はAI着手後の盤面になる。
- 直前のAI選択手や探索統計は含まれない。
- localStorageには追加されない。
- AI判断とは関係なく盤面だけを共有したい場合に使う。

### 4.2 AIの手を記録

直前のAI判断をブラウザ内の記録箱へ1件追加する。

保存対象:

- AIが考える前の盤面
- AIが選択した手
- AI難易度と評価profile
- 探索深度、node数、timeoutなどの探索統計

この操作だけではファイルは作成されない。後で調べたいAI手が出るたびに押し、後から「記録を保存」でまとめて取り出す。

### 4.3 記録を保存

localStorageへ蓄積した全記録をJSON配列として保存する。

ファイル名:

```text
bao-ai-review-YYYYMMDD-HHMMSS.json
```

ファイル保存後もlocalStorage内の記録は残る。保存しただけで記録が消えることはない。不要になった場合は別途「記録を削除」を使う。

推奨手順:

```text
後で調べたいAI着手が出る
  ↓
「AIの手を記録」を押す
  ↓
別の局面でも記録する
  ↓
3件程度集まったら「記録を保存」を押す
  ↓
日時付きJSONファイルを保管する
```

## 5. 初心者でも記録できる事象

次のいずれかを観測した場合は、最善手を判断できなくても記録してよい。

- AIが直後または数手以内に負けた。
- 大量捕獲の直後に、それ以上の反撃を受けた。
- 着手後に合法手や可動性が大きく減った。
- namuaからmtajiへ移った直後に一方的な形になった。
- 同じ局面で深度を変えると選択手が変わった。
- 同じ条件で繰り返すと選択手が安定しなかった。
- timeout、探索打ち切り、極端な長考が発生した。
- ルール上の勝利、強制手、即時敗北がプログラム上確認できた。
- AIの手の理由が分からず、後で調べたいと思った。

対局に負けたという事実だけではなく、疑問を持った特定のAI着手を記録する。

## 6. 保存したJSON

単一局面ファイルと記録配列ファイルは、どちらも`diagnostic-to-fixture.js`で読み込める。

記録配列は概ね次の構造を持つ。

```json
[
  {
    "format": "bao-ai-diagnostic",
    "version": 1,
    "position": {},
    "ai": {
      "level": "hard",
      "profile": "bao",
      "move": {},
      "stats": {}
    },
    "reason": "unexpected-ai-move"
  }
]
```

実際のJSONは手作業で変更せず、そのまま保存する。

## 7. Phase 10A：機械的局面監査

### 7.1 局面を再現する

各記録について次を確認する。

- ルールエンジン上で局面を復元できる。
- 観測されたAI手を再現できる。
- 合法手一覧を取得できる。
- phase、reserve、nyumba状態が保存されている。

再現できない記録は正式な候補にせず、診断形式または復元処理の問題として分離する。

### 7.2 探索深度を比較する

同じ局面を最低限、次の深度で比較する。

- 深度2
- 深度4
- 深度6以上
- 実行可能なら深度8

記録する内容:

- 各深度の選択手
- root評価値
- 上位候補手と評価差
- node数
- 到達深度
- timeout
- rootBestChangesなどの安定性指標

### 7.3 客観的な抽出条件

| 観測結果 | 暫定分類 |
| --- | --- |
| 深度を上げると選択手が変わる | 探索差あり |
| 深度を上げるほど評価が大きく悪化する | 浅い探索の錯覚候補 |
| 深度によって最善手が安定しない | 難局または探索効率候補 |
| 1位と2位の評価差が小さい | 最善手断定不可 |
| 選択後に可動性が急減する | 評価不足候補 |
| 選択後に大きな反撃を許す | 反撃評価候補 |
| 深度を上げても同じ手を選ぶ | 深い探索でも未解決、またはその手が妥当 |
| 再現実行ごとに手が変わる | 非決定性または時間制御候補 |

この段階では「探索不足」「評価不足」を確定しなくてよい。人間の期待手がない場合は`探索差あり`、`評価不足候補`、`難局`、`未判定`などを使う。

### 7.4 自己対局による相対比較

候補手が複数ある場合は、各着手後の局面から同条件で自己対局を行う。

確認する項目:

- 勝率
- 先手・後手別成績
- 平均手数
- timeout率
- 平均到達深度
- namuaとmtaji別成績

自己対局の結果だけでBao上の正解を確定しない。候補の優先順位付けと退行検出に使う。

## 8. fixtureの雛形へ変換する

プロジェクトルートで次を実行する。

```sh
node tools/diagnostic-to-fixture.js \
  --input /path/to/bao-ai-review-YYYYMMDD-HHMMSS.json \
  --output /tmp/bao-tactical-fixtures.js
```

配列内の1件だけを変換する場合は`--index`を使う。indexは0から始まる。

```sh
node tools/diagnostic-to-fixture.js \
  --input /path/to/bao-ai-review-YYYYMMDD-HHMMSS.json \
  --index 0 \
  --output /tmp/bao-tactical-case.js
```

生成される雛形は意図的に失敗するTODO assertionを含む。期待手と理由を確認していない局面が正式な回帰として通過することを防ぐためである。

## 9. レビュー状態

| 状態 | 意味 |
| --- | --- |
| `unreviewed` | 機械的記録のみ。棋力判断なし |
| `provisional` | 仮説はあるが、根拠または確認が不足 |
| `machine-supported` | 深度比較や自己対局が特定候補を支持 |
| `expert-reviewed` | 熟練者または十分な棋力根拠で確認済み |

`machine-supported`は`expert-reviewed`と同義ではない。

## 10. Phase 10B：棋力レビュー

各局面について次を確認する。

| 項目 | 確認内容 |
| --- | --- |
| 観測AI手 | AIが実際に選んだ手 |
| 問題点 | その手の何が不自然または不利なのか |
| 期待手 | 選ぶべき手。断定できなければ空欄 |
| 避ける手 | 期待手を1つに絞れない場合の禁止候補 |
| Bao上の理由 | 捕獲、反撃、前列、nyumba、reserve、可動性など |
| 判定方法 | 厳密な期待手、禁止手、勝利、形の維持など |

確定分類:

| 分類 | 条件 | 主な対応 |
| --- | --- | --- |
| 探索不足 | 深度を上げると根拠確認済みの妥当手を選ぶ | 必要深度、探索効率、時間配分の回帰候補 |
| 評価不足 | 深く探索しても根拠確認済みの不適切手を選ぶ | 同種反例を集め、Phase 12候補 |
| 期待手の誤認 | 人間側の期待手に十分な根拠がない | 正式回帰へ採用しない |
| 妥当手 | 観測AI手に問題がないと確認できた | 問題局面から除外する |
| レビュー待ち | 結論に必要な棋力または根拠が不足 | 状態付きで保持する |

## 11. 正式な戦術回帰へ追加する条件

レビュー済みfixtureのTODOを、局面意図に合ったassertionへ置き換える。

採用可能な判定例:

- 期待手と一致する。
- 特定の悪手を選ばない。
- 即時または強制勝利へ進む。
- 相手の最大反撃量を一定以下に抑える。
- mtaji移行後の前列占有や可動性を維持する。

複数の妥当な手がある場合は、特定手の厳密一致ではなく局面の本質を表す条件を使う。

追加後は次を実行する。

```sh
node test/tactical.test.js
```

## 12. Phase 10Aの完了条件

- 異なる局面を目安として3件以上保存した。
- 日時付きJSONファイルとして取り出した。
- 各局面をルールエンジン上で再現できた。
- 観測AI手と合法手一覧を確認した。
- 深度2、4、6以上を比較した。
- 客観的な抽出理由を記録した。
- 各局面へレビュー状態を付けた。
- 根拠未確認のfixtureを正式な戦術回帰へ追加していない。

期待手またはBao上の理由が空欄でも、Phase 10Aは完了できる。

## 13. Phase 12への移行条件

Phase 10Aの機械的異常だけではPhase 12へ進まない。次をすべて満たす場合だけ評価特徴量を検討する。

- 同種の反例が複数局面で再現する。
- 深度を上げても問題が解消しない。
- Phase 10Bで観測AI手の問題点とBao上の理由が確認されている。
- 既存評価内訳では失敗を説明できない、または特定特徴量の不足を説明できる。
- 新特徴量で改善を判定できる回帰条件が定義できる。

## 14. Codexへ依頼する場合

```text
保存した bao-ai-review-YYYYMMDD-HHMMSS.json の各AI着手を再現し、
深度2・4・6以上で選択手、root評価、上位候補との差、node数、
timeout、着手安定性を比較してください。

Bao上の期待手は自動で決めず、機械的な暫定分類とレビュー状態を付けてください。
根拠未確認の局面は正式な戦術回帰へ追加しないでください。
```
