# Bao AI 局面監査・対人レビュー手順

Version: 0.2.0
更新日: 2026-07-13

## 1. 目的

実際の対局で得たAI着手を、再現可能な診断局面として保存し、探索改善、評価改善、戦術回帰の候補へ変換する。

この手順は、レビュー担当者がBaoの熟練者であることを前提としない。人間が「この手は弱い」「期待手はこれだ」と判断できない場合でも、観測事実、探索深度差、評価値、自己対局結果を記録することで少しずつ進められる。

Phase 10は次の2段階に分ける。

| 段階 | 目的 | Baoの高い棋力 |
| --- | --- | --- |
| Phase 10A: 機械的局面監査 | 局面保存、再現、深度比較、客観的異常の抽出 | 不要 |
| Phase 10B: 棋力レビュー | 期待手、避ける手、Bao上の理由を確認し、戦術回帰へ採用する | 必要または外部確認が必要 |

Phase 10Aで未判定の局面が残っても問題はない。根拠が不十分な局面は、正式な戦術回帰や評価関数変更へ使わず、将来のレビュー候補として保持する。

## 2. 基本原則

- 初心者は最善手を断定しない
- 「弱そう」という印象より、観測できた事象を記録する
- 未判定を失敗扱いしない
- 期待手とBao上の理由が未確認のfixtureは正式な回帰へ追加しない
- 1局だけを根拠に評価関数を変更しない
- 自己対局は絶対的な正解判定ではなく、退行検出と候補比較に使う
- 熟練者レビューは、自動診断で絞り込んだ少数局面へ集中させる

## 3. 保存とプライバシー

AI診断記録は利用者がボタンを押した場合だけ、ブラウザのlocalStorageへ保存される。

保存する情報:

- AIが考える前の盤面
- AIが選択した着手
- AI難易度と評価profile
- 探索深度、node数、timeoutなどの許可された探索統計

保存しない情報:

- 自由記述
- 氏名、対局者名
- 端末やブラウザの識別子
- 保存時刻
- AI着手以外の連続操作ログ

記録は最大50件で、外部へ自動送信されない。「記録を削除」またはブラウザのサイトデータ削除機能から消去できる。

## 4. AI着手を記録する

### 4.1 対局条件

ゲームを開き、主に次の条件でAIと対局する。

- 難易度: 「むずかしい」または「ムタアラム」
- 手番: 可能なら先手と後手の両方
- phase: namuaとmtajiの両方

### 4.2 初心者でも記録できる事象

次のいずれかを観測した場合は、最善手を判断できなくても記録してよい。

- AIが直後または数手以内に負けた
- 大量捕獲の直後に、それ以上の反撃を受けた
- 着手後に合法手や可動性が大きく減った
- namuaからmtajiへ移った直後に一方的な形になった
- 同じ局面で深度を変えると選択手が変わった
- 同じ条件で繰り返すと選択手が安定しなかった
- timeout、探索打ち切り、極端な長考が発生した
- ルール上の勝利、強制手、即時敗北がプログラム上確認できた
- AIの手の理由が分からず、後で調べたいと思った

対局に負けたという事実だけではなく、疑問を持った特定のAI着手を記録する。

### 4.3 記録操作

1. 記録したいAI着手の直後に、画面下部の「AI改善用診断」を開く
2. 「AIの手を記録」を押す
3. 「AI着手を保存しました」という表示を確認する
4. 異なる局面で同じ操作を行う

最初は最低3件を目安にする。ただし、3件すべてについて期待手や問題点を説明できる必要はない。

### 4.4 診断ボタンの違い

| ボタン | 対象 | 保存先 | 主な用途 |
| --- | --- | --- | --- |
| 現在局面をコピー | 現在の盤面 | クリップボード | 盤面だけを1件、その場で共有する |
| AIの手を記録 | 直前のAI着手前の盤面、選択手、探索統計 | ブラウザ内のlocalStorage | 後で調べたいAI判断を記録箱へ追加する |
| 記録をコピー | 蓄積した全記録 | クリップボード | 複数のAI判断をJSON配列として取り出す |

#### 現在局面をコピー

ボタンを押した時点の盤面を、単一の診断snapshotとしてコピーする。

- AI着手後に押した場合はAI着手後の盤面になる
- 直前のAI選択手や探索統計は含まれない
- localStorageには追加されない
- AI判断とは関係なく盤面だけを共有したい場合に使う

#### AIの手を記録

直前のAI判断をブラウザ内の記録箱へ1件追加する。

- AIが考える前の盤面
- AIが選択した手
- AI難易度と評価profile
- 探索深度、node数、timeoutなどの探索統計

この操作だけではクリップボードへコピーされない。後で調べたいAI手が出るたびに押し、後からまとめて取り出す。

#### 記録をコピー

localStorageへ蓄積した全記録をJSON配列としてクリップボードへコピーする。コピー後も端末内の記録は残る。

推奨手順:

```text
後で調べたいAI着手が出る
  ↓
「AIの手を記録」を押す
  ↓
別の局面でも記録する
  ↓
3件程度集まったら「記録をコピー」を押す
  ↓
JSONファイルへ保存する
```

## 5. 記録JSONを取り出す

「記録をコピー」を押し、コピー内容を次のようなファイルへ保存する。

```text
artifacts/human-review-01.json
```

JSONは次のような構造を持つ。

```json
[
  {
    "format": "bao-ai-diagnostic",
    "version": 1,
    "position": {},
    "ai": {
      "level": "hard",
      "profile": "bao",
      "move": {},
      "stats": {}
    },
    "reason": "unexpected-ai-move"
  }
]
```

実際のJSONは編集せず、そのまま保存する。

## 6. Phase 10A：機械的局面監査

### 6.1 局面を再現する

各記録について、次を確認する。

- ルールエンジン上で局面を復元できる
- 観測されたAI手を再現できる
- 合法手一覧を取得できる
- phase、reserve、nyumba状態が保存されている

再現できない記録は正式な候補にせず、診断形式または復元処理の問題として分離する。

### 6.2 探索深度を比較する

同じ局面を最低限、次の深度で比較する。

- 深度2
- 深度4
- 深度6以上
- 実行可能なら深度8

記録する内容:

- 各深度の選択手
- root評価値
- 上位候補手と評価差
- node数
- 到達深度
- timeout
- rootBestChangesなどの安定性指標

### 6.3 客観的な抽出条件

| 観測結果 | 暫定分類 |
| --- | --- |
| 深度を上げると選択手が変わる | 探索差あり |
| 深度を上げるほど評価が大きく悪化する | 浅い探索の錯覚候補 |
| 深度によって最善手が安定しない | 難局または探索効率候補 |
| 1位と2位の評価差が小さい | 最善手断定不可 |
| 選択後に可動性が急減する | 評価不足候補 |
| 選択後に大きな反撃を許す | 反撃評価候補 |
| 深度を上げても同じ手を選ぶ | 深い探索でも未解決、またはその手が妥当 |
| 再現実行ごとに手が変わる | 非決定性または時間制御候補 |

この段階では「探索不足」「評価不足」を確定しなくてよい。人間の期待手がない状態では、`探索差あり`、`評価不足候補`、`難局`、`未判定`などの暫定分類を使う。

### 6.4 自己対局による相対比較

候補手が複数ある場合は、各着手後の局面から同条件で自己対局を行う。

確認する項目:

- 勝率
- 先手・後手別成績
- 平均手数
- timeout率
- 平均到達深度
- namuaとmtaji別成績

自己対局の結果だけでBao上の正解を確定しない。候補の優先順位付けと退行検出に使う。

## 7. 戦術fixtureの雛形へ変換する

プロジェクトルートで次を実行する。

```sh
node tools/diagnostic-to-fixture.js \
  --input artifacts/human-review-01.json \
  --output /tmp/bao-tactical-fixtures.js
```

配列内の1件だけを変換する場合は`--index`を使う。

```sh
node tools/diagnostic-to-fixture.js \
  --input artifacts/human-review-01.json \
  --index 0 \
  --output /tmp/bao-tactical-case.js
```

生成される雛形には、意図的に失敗するTODO assertionが残る。

```js
{
  category: "human-review",
  name: "TODO: explain reviewed diagnostic 1",
  depth: 4,
  position: state({ /* 再現局面 */ }),
  // Observed AI move: {...}
  // TODO: confirm the expected or forbidden move with a Bao reviewer.
  // TODO: explain whether deeper search solves it or evaluation knowledge is required.
  assert(analysis, position) {
    assert.fail("TODO: define the reviewed tactical expectation");
  },
}
```

Phase 10Aではfixtureの雛形作成まで進めてもよいが、期待手と理由が未確認なら`test/tactical.test.js`へ正式追加しない。

## 8. Phase 10B：棋力レビュー

棋力レビューでは、各局面について次を確認する。

| 項目 | 確認内容 |
| --- | --- |
| 観測AI手 | AIが実際に選んだ手 |
| 問題点 | その手の何が不自然または不利なのか |
| 期待手 | 選ぶべき手。断定できなければ空欄にする |
| 避ける手 | 期待手を1つに絞れない場合に示す |
| Bao上の理由 | 捕獲、反撃、前列、nyumba、reserve、可動性など |
| 判定方法 | 厳密な期待手、禁止手、勝利、形の維持など |
| 根拠水準 | 未確認、暫定、機械的裏付けあり、熟練者確認済み |

期待手に確信がない場合は、無理にテストへ追加せずレビュー待ちとして保持する。

レビュー状態は次を使う。

| 状態 | 意味 |
| --- | --- |
| `unreviewed` | 機械的監査のみ完了 |
| `provisional` | 人間の暫定判断がある |
| `machine-supported` | 深度比較や自己対局による裏付けがある |
| `expert-reviewed` | 熟練者が期待手と理由を確認した |

## 9. 熟練者レビューへ回す局面の優先順位

すべての記録を熟練者へ依頼する必要はない。次を優先する。

1. 深度を上げても悪い結果が繰り返される局面
2. 同種の問題が複数局面で発生するもの
3. 自己対局の勝敗を大きく左右するもの
4. 評価関数変更の候補になり得るもの
5. nyumba、namua、mtajiなどBao固有知識が必要なもの

熟練者へは、盤面だけでなく次の情報を添える。

- 深度ごとの選択手
- 各候補の評価値
- 自己対局結果
- AIが実際に選んだ手
- 確認したい論点

## 10. 戦術回帰へ追加する

レビュー済みfixtureのTODOを、局面意図に合ったassertionへ置き換える。

採用可能な判定例:

- 期待手と一致する
- 特定の悪手を選ばない
- 即時または強制勝利へ進む
- 相手の最大反撃量を一定以下に抑える
- mtaji移行後の前列占有や可動性を維持する
- 一定深度以上で選択手が安定する

複数の妥当な手がある場合は、期待手の厳密一致ではなく局面の本質を表す条件を使う。

追加後は次を実行する。

```sh
node test/tactical.test.js
```

必要に応じて診断出力も確認する。

```sh
BAO_TACTICAL_DIAG=1 node test/tactical.test.js
```

## 11. 1件ごとの記録テンプレート

```text
### 局面ID:

- JSON index:
- phase:
- AI難易度:
- 観測AI手:
- 深度2の手:
- 深度4の手:
- 深度6の手:
- 深度8の手:
- 各手の評価値:
- node数・timeout:
- 選択後の自己対局結果:
- 客観的な抽出理由:
- 問題点: 未判定可
- 期待手または避ける手: 未判定可
- Bao上の理由: 未判定可
- 暫定分類: 探索差あり / 評価不足候補 / 難局 / 非決定性 / 未判定
- レビュー状態: unreviewed / provisional / machine-supported / expert-reviewed
- 戦術回帰への採用: 採用 / 保留 / 不採用
```

## 12. 完了条件

### Phase 10A：機械的局面監査

次を満たせば完了とする。

- AI着手を3件以上保存した
- 記録JSONをファイルとして取り出した
- 各局面をルールエンジン上で再現できた
- 観測AI手を確認した
- 深度2、4、6以上を比較した
- 客観的な抽出理由を記録した
- 暫定分類とレビュー状態を付けた
- 根拠がない局面を正式な戦術回帰へ追加していない

期待手、避ける手、Bao上の理由はPhase 10Aの必須条件ではない。

### Phase 10B：棋力レビュー

次を満たした局面だけを正式な戦術回帰候補にする。

- 期待手または避ける手に根拠がある
- Bao上の理由を説明できる
- 判定方法をassertionとして表現できる
- 深度比較または自己対局結果と矛盾しない
- 必要に応じて熟練者が確認した

評価不足の同種反例が複数あり、期待手と理由が確認できた場合に限りPhase 12の特徴量候補へ進める。

## 13. Codexへ依頼する場合

Phase 10Aの依頼例:

```text
artifacts/human-review-01.json の各AI着手を再現し、深度2・4・6・可能なら8で比較してください。
各深度の選択手、評価値、node数、timeout、最善手の安定性を記録し、
探索差あり・評価不足候補・難局・非決定性・未判定へ暫定分類してください。
Bao上の期待手は自動決定せず、根拠がない局面はunreviewedのまま保持してください。
```

Phase 10Bの依頼例:

```text
machine-supportedまたはexpert-reviewedの局面だけを対象に、
期待手または避ける手とBao上の理由を確認し、戦術回帰のassertion案を作成してください。
根拠が不十分な局面は回帰へ追加しないでください。
```

Codexは局面再現、fixture変換、深度比較、評価内訳、自己対局比較を行える。ただし、Bao上の期待手が妥当か不明な場合は自動決定せず、レビュー待ちとして残す。
