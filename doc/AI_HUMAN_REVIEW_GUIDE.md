# Bao AI 対人局面レビュー手順

Version: 0.1.0
更新日: 2026-07-11

## 1. 目的

実際の対局で「このAI着手は弱そう」「不自然に見える」「理由が分からない」と感じた局面を収集し、再現可能な戦術回帰または評価改善の根拠へ変換する。

Phase 10では、違和感のあるAI着手を最低3件集め、次のいずれかへ分類する。

| 分類 | 意味 | 主な次の対応 |
| --- | --- | --- |
| 探索不足 | 深度を上げると期待手を選べる | 探索回帰、探索効率、時間配分を検討する |
| 評価不足 | 深く探索しても不自然な手を選ぶ | Phase 12の評価特徴量候補として検討する |
| 期待手の誤認 | 人間側の期待手に十分な根拠がない | 回帰へ採用せず、レビュー待ちにする |

最低3件は統計的な十分条件ではない。同じ種類の問題が複数局面で再現するかを確認し、偶然の1局だけを根拠に評価関数を変更しないための入口である。

## 2. 保存とプライバシー

AI診断記録は利用者がボタンを押した場合だけ、ブラウザのlocalStorageへ保存される。

保存する情報:

- AIが考える前の盤面
- AIが選択した着手
- AI難易度と評価profile
- 探索深度、node数、timeoutなどの許可された探索統計

保存しない情報:

- 自由記述
- 氏名、対局者名
- 端末やブラウザの識別子
- 保存時刻
- AI着手以外の連続操作ログ

記録は最大50件で、外部へ自動送信されない。「記録を削除」またはブラウザのサイトデータ削除機能から消去できる。

## 3. AI着手を記録する

### 3.1 対局条件

ゲームを開き、主に次の条件でAIと対局する。

- 難易度: 「むずかしい」または「ムタアラム」
- 手番: 可能なら先手と後手の両方
- phase: namuaとmtajiの両方

記録候補の例:

- 大量捕獲を選び、その後に大きな反撃を受ける
- nyumbaを不自然に使用または温存する
- namuaからmtajiへ移る盤面形が悪い
- mtajiで前列や可動性を失う
- 明らかに見える勝ち筋を逃す
- すぐ負ける手や、敗北を早める手を選ぶ

対局に負けたという理由だけでは記録せず、疑問を持った特定のAI着手を記録する。

### 3.2 記録操作

1. AIが疑問のある手を指した直後に、画面下部の「AI改善用診断」を開く
2. 「AIの手を記録」を押す
3. 「AI着手を保存しました」という表示を確認する
4. 異なる局面で同じ操作を行い、最低3件集める

同じ対局中でも複数回記録できる。ただし、ほぼ同じ局面だけに偏らないようにする。

推奨する最初の3件:

| 件 | 局面の例 |
| --- | --- |
| 1 | namuaまたはnyumbaの判断に違和感がある局面 |
| 2 | 捕獲後の反撃が問題になる局面 |
| 3 | mtajiで前列維持や可動性が問題になる局面 |

## 4. 記録JSONを取り出す

最低3件を保存したら、「AI改善用診断」の「記録をコピー」を押す。クリップボードへ診断snapshotの配列がコピーされる。

コピーした内容を、例えば次のファイルへ保存する。

```text
artifacts/human-review-01.json
```

JSONは次のような構造を持つ。

```json
[
  {
    "format": "bao-ai-diagnostic",
    "version": 1,
    "position": {},
    "ai": {
      "level": "hard",
      "profile": "bao",
      "move": {},
      "stats": {}
    },
    "reason": "unexpected-ai-move"
  }
]
```

盤面配列などは上の例では省略している。実際のJSONは編集せず、そのまま保存する。

誤った記録をすべて消す場合は「記録を削除」を押す。現在は個別削除ではなく全件削除である。

## 5. 戦術fixtureの雛形へ変換する

プロジェクトルートで次を実行する。

```sh
node tools/diagnostic-to-fixture.js \
  --input artifacts/human-review-01.json \
  --output /tmp/bao-tactical-fixtures.js
```

配列内の1件だけを変換する場合は`--index`を使う。indexは0から始まる。

```sh
node tools/diagnostic-to-fixture.js \
  --input artifacts/human-review-01.json \
  --index 0 \
  --output /tmp/bao-tactical-case.js
```

3件目だけなら`--index 2`を指定する。

生成される雛形には、次のTODOが残る。

```js
{
  category: "human-review",
  name: "TODO: explain reviewed diagnostic 1",
  depth: 4,
  position: state({ /* 再現局面 */ }),
  // Observed AI move: {...}
  // TODO: confirm the expected or forbidden move with a Bao reviewer.
  // TODO: explain whether deeper search solves it or evaluation knowledge is required.
  assert(analysis, position) {
    assert.fail("TODO: define the reviewed tactical expectation");
  },
}
```

このassertionは意図的に失敗する。期待手と理由を確認していない局面が、正式な回帰テストとして通過することを防ぐためである。

## 6. 人間によるレビュー

各局面について、次の項目を記録する。

| 項目 | 確認内容 |
| --- | --- |
| 観測AI手 | AIが実際に選んだ手 |
| 問題点 | その手の何が不自然または不利なのか |
| 期待手 | 選ぶべき手。断定できなければ空欄にする |
| 避ける手 | 期待手を1つに絞れない場合、避けるべき手を示す |
| Bao上の理由 | 捕獲、反撃、前列、nyumba、reserve、可動性など |
| 判定方法 | 厳密な期待手、禁止手、勝利、形の維持など |

期待手に確信がない場合は、無理にテストへ追加せずレビュー待ちとして保持する。

## 7. 探索深度による診断

同じ局面を最低限、次の深度で比較する。

- 深度2
- 深度4
- 深度6以上

診断の目安:

| 観測結果 | 分類 |
| --- | --- |
| 深度を上げると期待手を選ぶ | 探索不足 |
| 深度を上げても同じ不自然な手を選ぶ | 評価不足の可能性 |
| 深度によって最善手が安定しない | 難局または探索効率の問題 |
| 人間の期待手を説明できない | 期待手の誤認またはレビュー待ち |

探索不足と判断した局面は、必要深度を守る探索回帰候補にする。評価不足と判断した局面は、同種の反例が複数ある場合だけPhase 12の特徴量候補へ進める。

## 8. 戦術回帰へ追加する

レビュー済みfixtureのTODOを、局面意図に合ったassertionへ置き換える。

採用可能な判定例:

- 期待手と一致する
- 特定の悪手を選ばない
- 即時または強制勝利へ進む
- 相手の最大反撃量を一定以下に抑える
- mtaji移行後の前列占有や可動性を維持する

必ずしも期待手の厳密一致を使う必要はない。複数の妥当な手がある場合は、局面の本質を表す条件を使う。

追加後は次を実行する。

```sh
node test/tactical.test.js
```

必要に応じて診断出力も確認する。

```sh
BAO_TACTICAL_DIAG=1 node test/tactical.test.js
```

## 9. 1件ごとの記録テンプレート

```text
### 局面ID:

- JSON index:
- phase:
- AI難易度:
- 観測AI手:
- 問題点:
- 期待手または避ける手:
- Bao上の理由:
- 深度2の手:
- 深度4の手:
- 深度6以上の手:
- 分類: 探索不足 / 評価不足 / 期待手の誤認 / レビュー待ち
- 戦術回帰への採用: 採用 / 保留 / 不採用
```

## 10. 完了条件

「AI着手を最低3件記録」は、次をすべて満たしたときに完了とする。

- 違和感のあるAI着手を3件以上保存した
- 記録JSONをファイルとして取り出した
- 各局面をルールエンジン上で再現できた
- 観測AI手を確認した
- 期待手または避ける手を検討した
- Bao上の理由を記述した
- 深度2、4、6以上を比較した
- 探索不足、評価不足、期待手の誤認、レビュー待ちのいずれかへ分類した
- 根拠が十分な局面だけを戦術回帰へ追加した

最低3件を分類した後、評価不足の同種反例が複数ある場合に限りPhase 12へ進む。

## 11. Codexへ依頼する場合

記録JSONをワークスペースへ置き、次のように依頼する。

```text
artifacts/human-review-01.json のAI着手を、深度2・4・6以上で比較し、
探索不足・評価不足・期待手の誤認へ分類してください。
期待手に根拠がない局面は回帰へ追加しないでください。
```

Codexは局面再現、fixture変換、深度比較、評価内訳の確認を行える。ただし、Bao上の期待手が妥当か不明な場合は自動決定せず、人間のレビューを求める。
