Version: 0.3.0  
Status: Active  
作成日: 2026-07-07  
更新日: 2026-07-13

この文書は、`AI_ROADMAP.md`のPhase 0〜5完了後に進める追加強化計画である。正式な測定条件と基準成績は`AI_BENCHMARK.md`、設計判断、試行錯誤、既知の限界は`AI_DEVELOPMENT_LOG.md`へ記録する。

具体的な局面保存、深度比較、暫定分類、棋力レビューの操作手順は`AI_HUMAN_REVIEW_GUIDE.md`を正とする。

## 1. 目的

ブラウザ上で軽量に動作する現在の構成を維持しながら、Bao AIの探索効率、局面評価、戦術的な信頼性を段階的かつ再現可能な方法で改善する。

Phase 5でMCTSを既定AIへ採用しなかったため、主軸は次に置く。

- 既存Alpha-Beta系探索の効率化
- Bao固有評価の反例駆動改善
- 固定戦術局面による回帰防止
- seed付き自己対局による変更前後比較
- 実対局由来の局面を用いた機械的監査と棋力レビュー

## 1.1 強化アプローチ

このプロジェクトは、軽量なブラウザ実行を前提にした「古典的ゲームAI + Bao固有のドメイン知識 + 自己対局チューニング」を採用する。

| 要素 | 役割 |
| --- | --- |
| ルールエンジン | 合法手生成と局面遷移を一元的に保証する |
| 評価関数 | 捕獲、前列安全性、nyumba、reserve、namua/mtaji移行などを数値化する |
| Alpha-Beta系探索 | 評価関数だけでは見えない数手先の得失を読む |
| 戦術回帰 | 自己対局勝率だけでは見落とす局面品質を守る |
| 自己対局 | 候補比較、退行検出、重み調整に使う |
| 局面監査 | 実対局由来の局面を、棋力に依存しない観測情報として蓄積する |
| 棋力レビュー | 期待手、避ける手、Bao上の理由を確認し、正式な回帰へ昇格させる |
| Worker実行 | 強さを上げてもUI応答性を維持する |

大まかな流れ:

```text
ルールエンジン
  ↓
合法手生成
  ↓
Bao固有の局面評価
  ↓
Alpha-Beta系探索
  ↓
自己対局・戦術回帰・局面監査で検証
  ↓
根拠が確認できた変更だけを採用
```

## 3. 基本方針

1. 自己対局勝率だけで候補を採用しない。
2. 探索改善、評価変更、時間配分変更を同時に行わない。
3. 固定深度試験と時間制限試験を分ける。
4. 評価特徴量は、深い探索でも解けない反例が複数確認された場合だけ追加する。
5. 人間の対局ログを使う場合も、採用判断は再現可能な局面、測定値、回帰条件へ落とし込む。
6. Baoの習熟度が十分でないレビュー担当者へ、期待手や不自然さの断定を要求しない。
7. 未判定局面は削除せず、根拠の強さを示す状態付きで保持する。
8. 同じopeningを共有するpaired gameの相関を統計判定で考慮する。

## 8. Phase 10：実対局由来の改善ループ

目安: 中長期  
実施状況: 進行中（2026-07-13、10Aの実装基盤完了）

### 狙い

実際の対局で得たAI着手を再現可能な診断局面として保存し、棋力に依存しない機械的監査と、根拠確認後の棋力レビューを分離する。

レビュー担当者がBaoの最善手を判断できない場合でも、Phase 10Aは進められる。期待手やBao上の理由を無理に作らない。

### 8.1 共通前提

- 記録は利用者の明示操作時だけ行う。
- 自由記述、氏名、対局者名、端末識別子、保存時刻、連続操作ログを保存しない。
- 記録はlocalStorageへ最大50件保存し、外部へ自動送信しない。
- 未判定局面を正式な戦術回帰または評価変更の根拠として扱わない。
- 自己対局結果は相対比較に使い、Bao上の絶対的な正解とはみなさない。

### 8.2 実装済み基盤

- [x] 現在局面をJSONとしてコピーする機能
- [x] 直前AI着手前の盤面、選択手、探索統計を記録する機能
- [x] 記録JSONのコピーと全削除
- [x] version付き診断形式と局面復元
- [x] 単一snapshot、記録配列、index指定からfixture雛形を生成する入口
- [x] 未レビューfixtureが正式回帰として通過しないTODO assertion

### 8.3 Phase 10A：機械的局面監査

Baoの高い棋力を必要としない段階である。

#### 実施内容

1. 実対局由来のAI着手を記録する。
2. 局面、phase、reserve、nyumba状態、観測AI手を再現する。
3. 深度2、4、6以上、可能なら8で同じ局面を比較する。
4. 各深度の選択手、root評価、上位候補との差、node数、到達深度、timeout、着手安定性を記録する。
5. 必要に応じて候補手後の局面から同条件で自己対局する。
6. 客観的な抽出理由とレビュー状態を記録する。

#### 暫定分類

| 観測結果 | 暫定分類 |
| --- | --- |
| 深度を上げると選択手が変わる | 探索差あり |
| 深度を上げるほど評価が悪化する | 浅い探索の錯覚候補 |
| 深度ごとに最善手が安定しない | 難局または探索効率候補 |
| 上位候補の評価差が小さい | 最善手断定不可 |
| 着手後に可動性が急減する | 評価不足候補 |
| 着手後に大きな反撃を許す | 反撃評価候補 |
| 深度を上げても同じ手を選ぶ | 深い探索でも未解決、または選択手が妥当 |
| 再現実行ごとに手が変わる | 非決定性または時間制御候補 |

この段階では「探索不足」「評価不足」「期待手の誤認」を確定しない。人間の期待手がない場合は`未判定`のまま保持する。

#### レビュー状態

| 状態 | 意味 |
| --- | --- |
| `unreviewed` | 機械的記録のみ。棋力判断なし |
| `provisional` | 仮説はあるが、根拠または確認が不足 |
| `machine-supported` | 深度比較や自己対局が特定候補を支持 |
| `expert-reviewed` | 熟練者または十分な棋力根拠で確認済み |

`machine-supported`は`expert-reviewed`と同義ではない。

#### Phase 10Aの完了条件

- 異なる局面を目安として3件以上保存した。
- JSONをファイルとして取り出した。
- 各局面をルールエンジン上で再現できた。
- 観測AI手と合法手一覧を確認した。
- 深度2、4、6以上を比較した。
- 客観的な抽出理由を記録した。
- 各局面へレビュー状態を付けた。
- 根拠未確認のfixtureを正式な戦術回帰へ追加していない。

3件は統計的十分条件ではなく、運用を開始するための目安である。期待手またはBao上の理由が空欄でも、Phase 10Aは完了できる。

### 8.4 Phase 10B：棋力レビュー

Bao上の期待手、避ける手、局面理由を確認する段階である。担当者の習熟後、外部熟練者の協力時、または十分な根拠が得られた時点で実施する。

#### 実施内容

- 観測AI手の問題点をBao上の概念で説明する。
- 期待手または避ける手を確認する。
- 捕獲、反撃、前列、nyumba、reserve、可動性、namua/mtaji移行などの理由を記録する。
- 深度差と棋力判断を組み合わせ、探索不足、評価不足、期待手の誤認へ確定分類する。
- 複数の妥当手がある場合は、特定手一致ではなく局面の本質を表す条件を使う。

#### 確定分類

| 分類 | 条件 | 主な対応 |
| --- | --- | --- |
| 探索不足 | 深度を上げると根拠確認済みの妥当手を選ぶ | 必要深度、探索効率、時間配分の回帰候補 |
| 評価不足 | 深く探索しても根拠確認済みの不適切手を選ぶ | 同種反例を集め、Phase 12候補 |
| 期待手の誤認 | 人間側の期待手に十分な根拠がない、または機械比較が支持しない | 正式回帰へ採用しない |
| 妥当手 | 観測AI手に問題がないと確認できた | 問題局面から除外し、必要なら学習記録として保持 |
| レビュー待ち | 結論に必要な棋力または根拠が不足 | 状態付きで保持する |

#### Phase 10Bの完了条件

- 期待手または避ける手の根拠を確認した。
- Bao上の理由を記述した。
- 機械診断と棋力判断を分離して記録した。
- 根拠に応じてレビュー状態を更新した。
- 正式回帰へ追加する場合は、判定条件と必要探索深度を明記した。
- 回帰追加後に自己対局、既存戦術回帰、UI性能を再確認した。

Phase 10BはPhase 10Aの全局面に対して完了させる必要はない。自動診断で重要度の高い少数局面へ集中する。

### 8.5 Phase 12への移行条件

Phase 10Aの機械的異常だけではPhase 12へ進まない。次をすべて満たす場合だけ評価特徴量を検討する。

- 同種の反例が複数局面で再現する。
- 深度を上げても問題が解消しない。
- Phase 10Bで観測AI手の問題点とBao上の理由が確認されている。
- 既存評価内訳では失敗を説明できない、または特定特徴量の不足を説明できる。
- 新特徴量で改善を判定できる回帰条件が定義できる。

## 9. Phase 11：探索効率 v2

目安: 1〜2週間  
実施状況: 完了（2026-07-11）

### 狙い

評価関数と固定探索予算を変えず、着手順序、Transposition Table、評価再利用を個別に比較する。

### 結果

| 候補 | 判定 |
| --- | --- |
| 探索内評価キャッシュ | hard/expertへ採用。上限2,048 entries |
| TT最善手優先 | 一部改善したがholdoutで再現性不足。保留 |
| Quiescence捕獲順序 | 安定した実時間改善に届かず保留 |
| history heuristic | 追加効果がほぼなく不採用 |
| 固定幅aspiration window | 再探索によりnode増加。不採用 |
| TT終局距離正規化 | 利益を観測できず、正確性リスクのため不採用 |

### 採用条件

次のいずれかを満たし、既存戦術回帰を悪化させない候補だけを採用する。

- 同一固定深度・同一選択手で、代表局面の中央値node数または思考時間を10%以上削減する。
- 同一時間制限で平均到達深度を上げ、500局以上で現行探索に対する統計的優位を示す。
- 同等勝率を維持しながら、timeout率または最大思考時間を明確に下げる。

## 10. Phase 12：反例駆動の評価改善

目安: 中長期  
実施状況: 未着手

### 狙い

Phase 10Bで問題が確認され、深い探索でも解けない同種反例だけを出発点として、新しい評価特徴量を検討する。

Phase 10Aの`評価不足候補`は、Phase 12へ直接渡さない。棋力レビュー前は、観測AI手が実際には妥当である可能性を残す。

### 候補特徴量の例

- `replyCaptureRisk`: 着手後に許す相手の最大捕獲量
- `postRelaySafety`: relay終了後の前列占有、連結、再利用可能穴
- `forcedCaptureDepth`: 強制捕獲系列が続く長さ
- `houseSurvival`: nyumbaを相手の次手以降まで維持できる見込み
- `mobilityAfterReply`: 相手の有力な反撃後にも残る可動性

候補名は仮称である。実際の反例を説明できない特徴量は実装しない。

### 完了条件

- 導入理由となる根拠確認済みの反例が複数ある。
- 各反例に期待手、避ける手、または局面条件としての回帰判定がある。
- 反例局面で改善し、既存戦術回帰を悪化させない。
- 固定深度とブラウザ相当時間の両方で性能コストを記録する。
- 未使用seedとpaired openingを使う500局以上の最終確認、または戦術解決率の明確な改善を示す。

## 11. 推奨順序

Phase番号順ではなく、採用判断の依存関係を優先する。

1. Phase 10Aで実対局由来局面を保存し、再現と深度比較を行う。
2. 客観的な異常、着手安定性、評価差、自己対局結果から重要局面を絞る。
3. 未判定局面を`unreviewed`または`machine-supported`として保持する。
4. 習熟後または熟練者の協力時に、優先度の高い局面だけPhase 10Bで確認する。
5. 探索不足と確認された局面は探索回帰へ追加する。
6. 深い探索でも解けない評価不足が複数確認された場合だけPhase 12へ進む。
7. Phase 12で特徴量候補が得られた場合に限り、Phase 7またはPhase 9の再検証を行う。

既存の`bao-v2`補正、適応探索予算、重み再探索を、反例なしで初期候補へ混ぜない。

## 12. 採用判定

次のいずれかを満たす変更を採用候補とする。

- 500局以上で基準AIに統計的優位を示し、戦術回帰を維持する。
- 勝率は同等だが、根拠確認済み戦術局面の解決率が明確に上がる。
- UI応答性を維持したまま、同等勝率で平均または最大思考時間が下がる。
- 同一固定深度と同一選択手で、探索node数または思考時間が再現可能に減る。

次の場合は不採用または保留とする。

- 自己対局勝率だけ高く、戦術回帰を悪化させる。
- 時間制限試験だけで強く、固定深度または複数seedで再現しない。
- 未レビュー局面を正解として固定している。
- Cloudflare Pagesで公開する`public/`の安定性を下げる。
- 採用理由を局面または測定値で説明できない。

統計的優位の判定はWilson 95%下限を記録する。ただし、paired openingの2局には相関があるため、opening pair単位の勝点差またはbootstrap信頼区間も併記し、個別対局を完全に独立とみなした値だけで境界候補を採用しない。

## 14. 次の最小作業単位

Phase 10Aの実装基盤は完了している。次は棋力判断を要求しない小さな監査作業から開始する。

1. hardまたはexpertとの実対局で、後で調べたいAI着手を1件記録する。
2. 記録JSONを`artifacts/human-review-01.json`へ保存する。
3. 局面復元と観測AI手の再現を確認する。
4. 深度2、4、6以上で選択手と探索統計を比較する。
5. 客観的な抽出理由を記録し、状態を`unreviewed`とする。
6. 同じ手順を異なる局面で繰り返し、目安として3件集める。
7. 期待手やBao上の理由を説明できない局面は、そのまま将来のレビュー候補として保持する。

Phase 10Aを少しずつ進める間は、既存評価重みの再調整やPhase 12の特徴量実装を先行しない。Phase 10Bは、習熟度の向上または外部レビューの機会に応じて段階的に実施する。
