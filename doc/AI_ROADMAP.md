# Bao la Kiswahili コンピュータ強化ロードマップ

Version: 0.1.0  
Status: Completed

設計判断、試行錯誤、既知の限界は`AI_DEVELOPMENT_LOG.md`に記録する。Phase 0-5完了後の追加強化計画は`AI_ADVANCED_ROADMAP.md`を参照する。

## 1. 目的

ブラウザ上で快適に動作するという現在の要件を維持しながら、コンピュータ対戦の強さを段階的かつ測定可能な形で向上させる。

強化は、次の順序で進める。

1. 強さを測る基盤を整える
2. Bao固有の局面評価を改善する
3. 探索効率と探索深度を高める
4. UIを止めずに思考できるようにする
5. 自己対戦で評価値を調整する
6. 必要に応じてMCTSを比較検証する

## 2. 現状

`ai.js`には以下の3段階のAIが実装されている。

| 難易度 | 現在の方式 |
| --- | --- |
| やさしい | 合法手からランダム選択 |
| ふつう | 1手後の局面評価上位3手からランダム選択 |
| むずかしい | 反復深化、Minimax、Alpha-Beta枝刈り |

「むずかしい」は最大深度4、思考時間約450msである。評価関数は主に次の要素を使用している。

- 盤上の石数差
- 前列の占有穴数
- 前列の石数差
- nyumbaの所有状態
- 直後の捕獲数

現時点では、同一局面の再利用、探索統計、自己対戦ベンチマーク、Bao固有の詳細な戦術評価は実装されていない。

## 3. 基本方針

- ルール判定には常に既存の`engine.js`を使用する
- AIの変更前後を自動対戦の勝率で比較する
- 強さだけでなく、思考時間とUIの応答性を維持する
- 各フェーズを独立して検証可能にする
- MCTSは先に採用せず、改善済みAlpha-Beta探索との比較結果で判断する

## 5. Phase 1：評価関数のBao固有化

目安: 3〜5日

実施状況: 完了（2026-07-06）

評価項目ごとのテストと旧評価との200局比較結果は`AI_BENCHMARK.md`を参照する。

最も優先度の高いフェーズとする。単に探索を深くする前に、葉局面を正しく評価できるようにする。

### 追加する評価要素

- 次の手で捕獲できる石数
- 相手に許す最大捕獲数
- 自分と相手の捕獲可能手数
- 捕獲後に発生するrelay sowingの形
- 前列が空になる危険度
- 前列の連結性と再利用可能な穴
- 着地点と次手の捕獲可能性
- nyumbaの利用価値、維持価値、崩壊リスク
- reserveの残量と投入効率
- namua終了時に残る盤面の形
- mtajiにおける可動性
- 手番によるテンポ差

### 実装方針

- namuaとmtajiで評価項目の重みを分ける
- 評価項目を個別の関数に分割する
- 重みを一か所にまとめ、後から調整できるようにする
- 各評価項目に単体テストを用意する

### 完了条件

- 代表局面で期待する着手を安定して選択する
- 現行の「むずかしい」と200局対戦し、先後合計で65%以上の勝率を得る

## 6. Phase 2：探索アルゴリズムの改善

目安: 4〜7日

実施状況: 完了（2026-07-06）

探索性能とPhase 1探索との比較結果は`AI_BENCHMARK.md`、設計判断は`AI_DEVELOPMENT_LOG.md`を参照する。

### 実装内容

- 盤面を一意に表す局面キーを実装する
- Transposition Tableで同一局面の探索結果を再利用する
- 反復深化の前回最善手を次の探索で優先する
- 着手順序を次の優先順位で改善する
  1. 即時勝利
  2. 大きな捕獲
  3. 前回反復深化の最善手
  4. killer move
  5. 静的評価値
- Principal Variation Searchを検討する
- 捕獲が継続する局面向けにQuiescence Searchを追加する
- 勝敗評価に終局までの距離を反映する
  - より早い勝利を選ぶ
  - 避けられない敗北を遅らせる
- 探索ノード数、枝刈り数、キャッシュヒット率を計測する

### 注意点

- 局面キーには盤面だけでなく、手番、フェーズ、reserve、nyumba所有状態を含める
- 時間切れ時は、最後に完了した反復深化の結果を返す
- 不完全な探索結果をTransposition Tableへ確定値として保存しない

### 完了条件

- 450ms以内に探索できるノード数が現行比2倍以上になる
- 代表的な中盤局面で深度6以上へ到達する
- Phase 1のAIとの200局で60%以上の勝率を得る

## 7. Phase 3：Web Workerによる思考処理

目安: 2〜4日

実施状況: 完了（2026-07-06）

Worker構成、中止方式、端末別設定、統合テストは`AI_DEVELOPMENT_LOG.md`を参照する。

Phase 2時点のAI探索はメインスレッドで同期実行されるため、探索時間を延ばすと画面操作や描画が停止する。AIをWeb Workerへ分離し、UIと探索を独立させる。

### 実装内容

- AI探索をWeb Workerへ移動する
- 盤面、難易度、思考時間をメッセージで渡す
- 新規対局や画面遷移時に探索を中止できるようにする
- 端末性能に応じて探索深度を自動調整する
- 思考中表示を探索処理と同期する
- Workerが利用できない環境では現行方式へフォールバックする

### 推奨難易度

| 難易度 | 方針 | 思考時間の目安 |
| --- | --- | ---: |
| やさしい | ランダム中心 | 即時 |
| ふつう | 評価関数と意図的な揺らぎ | 50〜100ms |
| むずかしい | Alpha-Beta探索 | 400〜600ms |
| ムタアラム | 深い探索とTransposition Table | 1.5〜3秒 |

### 完了条件

- 「ムタアラム」の思考中も描画と操作が停止しない
- 探索の中止後に古い局面への着手が反映されない
- オフライン環境でもWorkerが正常に読み込まれる

## 8. Phase 4：自己対戦による評価値調整

目安: 1〜2週間

実施状況: 完了（2026-07-07）

重み分離、hill climbing、学習／検証seed分離、複数世代対戦、successive halving、複数seedによる候補選抜を実装した。複数seed版successive halvingで得た`namua.houseValue: -7`候補を500局検証し、Phase 3基準AIに対して統計的優位を確認した。詳細は`AI_BENCHMARK.md`と`AI_DEVELOPMENT_LOG.md`を参照する。

### 実装内容

- 評価関数の重みを設定データとして分離する
- 複数の重み候補を自動生成する
- 候補同士を先後交代で自己対戦させる
- hill climbingまたは遺伝的探索で重みを更新する
- 学習用局面と検証用局面を分離する
- 特定の開始局面だけに適応しないよう、途中局面からの対戦も行う

### 評価方法

- 勝率を主指標とする
- 先手・後手の性能差を別々に監視する
- 平均手数だけを強さの指標にしない
- 同一相手に対する過学習を避けるため、複数世代のAIと対戦する

### 完了条件

- Phase 3の基準AIとの500局で統計的に優位な成績を得る
- 戦術局面の回帰テストを悪化させない

## 9. Phase 5：MCTSの比較研究

目安: 中長期

実施状況: 完了（比較研究として不採用、2026-07-07）

Phase 5の入口として、既存の`analyzeMove`と自己対戦ベンチマークから呼び出せる実験用`mcts`探索プロファイルを追加した。MCTSはUCT、捕獲優先プレイアウト、評価関数終端、同一条件ベンチマークを試すための比較研究用の足場である。

2026-07-07時点の比較では、MCTS単体候補はPhase 2探索を上回っていない。Phase 2探索をroot候補制限に使う短いhybrid検証でも改善は確認できなかったため、Phase 5ではMCTSを不採用とする。

### 検証内容

- UCTによる着手選択
- ランダムではなくBaoの捕獲を優先するプレイアウト方策
- プレイアウト終端前の評価関数利用
- Alpha-BetaとMCTSのハイブリッド候補制限
- 同一思考時間での勝率、メモリ使用量、端末負荷の比較

### 準備済みの内容

- `searchProfile: "mcts"`を追加し、既存APIからMCTS実験を実行できるようにした
- MCTSの反復数、プレイアウト上限、探索係数をベンチマークCLIから指定できるようにした
- MCTSのプレイアウト方策を`random`、`capture`、`evaluation`で切り替えられるようにした
- `capture`方策では盤面から即時捕獲量を推定し、プレイアウト単価を下げた
- 非捕獲手の同点を減らす軽量な`balanced`方策を追加した
- root着手選択を`visits`と`value`で比較できるようにした
- root候補を静的評価上位N手へ絞る候補制限を追加した
- root候補を浅いPhase 2探索の上位N手へ絞るhybrid候補制限を追加した
- root子ノードへ静的評価の仮想訪問を入れる`static` priorを比較できるようにした
- 報酬設計を`evaluation`、`terminal`、`fast-terminal`で比較できるようにした
- `policy × root × reward`を複数seedで短く比較するグリッドCLIを追加した
- ベンチマーク結果へsimulation数とプレイアウト長を出力するようにした
- MCTSプロファイルが合法手を返し、seed付きrandom callbackで再現できることを回帰テストに追加した
- MCTS単体とhybrid候補制限の短い比較結果を保存し、不採用判断を記録した

### 比較手順

```sh
node bao-la-kiswahili/tools/benchmark.js --games 20 --seed 20260715 \
  --opening-plies 8 --first hard --second hard \
  --first-search mcts --second-search phase2 \
  --time-limit 450 --max-depth 8 \
  --mcts-playout-turns 80 --mcts-policy capture \
  --mcts-root visits --mcts-reward evaluation
```

短い確認では`--mcts-iterations`を指定し、時間ではなく反復数を固定する。比較ではAlpha-Betaと同じ思考時間で、勝率、simulation数、平均プレイアウト長、平均思考時間を併記した。

Baoでは1手のrelay sowingが長くなることがあり、プレイアウト単価が高くなる。今回の条件では、MCTS単体もhybrid候補制限もPhase 2探索を上回らなかった。最高AIの主軸は引き続きPhase 2からPhase 4までのAlpha-Beta系探索と評価関数改善に置く。

## 11. 推奨する最初の実装範囲

## 12. 継続的な品質基準

- `engine.test.js`と`ai.test.js`を常に通過させる
- AIが返すすべての着手を`engine.js`で合法手として検証する
- AIの強化とルール変更を同一コミットで混在させない
- 固定局面テストと自己対戦の双方を実行する
- 思考時間の上限を超えた場合でも合法手を返す
- Service Workerのキャッシュ対象へWorkerなどの追加ファイルを反映する
